Important note for pull requests

Pull requests should be issued against the develop branch. We never pull directly into master.
Our goal is to iterate in small steps. Release often, release early. Evolution instead of a revolution.

General goals

To make sure everybody is going in the same direction:
* easy to use for people
* using less resources
* extendable with plugins
* keep it maintainable

How to work with git?

* Don't work in your master branch.
* Make a new branch for every feature you are working on.
* Don't use the online edit function of github.
* Try to make clean commits that are easy readable (including descriptive commit messages).
* Test before you push.
* Make small pull requests that are easy to review but make sure they do add value individually.

Coding style

* Do write comments.
* Indentation: CSS: 2 spaces; HTML: 4 spaces (= 1 tab)
* Don't overengineer.
* Do generalize sooner then later.
* If you do make changes, document them. (see below)

Branching model / git workflow

Over the time we pull features from feature branches into the develop branch. Every month we pull from develop into master. Bugs in master get fixed in hotfix branches. These branches will get merged into master and develop. There should never be commits in master that aren't in develop.
Also see git workflow
# master branch
* This is the (stable) branch everyone should use for production stuff
# develop branch
* Everything that is tested and ready to go into master at some point in time
# release branch
* Only bugfixes go into these
# hotfix branch
* Fixes for bugs in master
# feature branch (in your own repos)
* These are the branches where you develop your features in
